**#2026-02-14**
**重构音视频录像架构，实现 1080P 30FPS 高清丝滑录像**
   - **遇到的问题**：
     - **录像加速/乱序**：由于 OpenCV 软编码（Software Encoding）极其消耗 CPU，导致 1080P 录像帧率严重不足，播放时出现“快进效应”且时间轴错乱（秒表顺序跳跃）。
     - **硬件加速兼容性**：尝试调用 RK3568 的 MPP 硬件加速插件，但因 OpenCV 内存类型与硬件 DMA 内存不匹配，导致 RGA 内存映射失败（`rga2_mmu: RGA2 failed to get pte`）。
   - **解决方案**：
     - **流直写技术（Stream Copy）**：放弃 OpenCV 录像，改为调用系统内置的 **FFmpeg** 引擎。通过 `-c:v copy` 参数直接封装 `mjpg-streamer` 已经压缩好的 JPEG 数据包。
     - **零负载录像**：该方案实现了 **0 CPU 负载、30FPS 满血录像**，彻底解决了预览与录像互相干扰的问题。

**#2026-02-24**
**OV5695 MIPI 摄像头底层驱动适配与格式攻坚**
   - **遇到的问题 1：设备不识别**。更换 MIPI 摄像头后，系统无法生成 `/dev/video0` 节点。排查发现引脚复用导致 I2C 地址冲突，且设备树（DTS）未正确使能。
   - **解决方案**：修改内核 **DTS (Device Tree Source)**，重新配置 `mipi_csi` 节点并解除引脚冲突，重新编译 **Kernel** 并烧录镜像，成功识别硬件。
   - **遇到的问题 2：软件不兼容**。`mjpg-streamer` 强行要求硬件输出 MJPEG 格式，但 OV5695 通过 RKISP 仅支持 **RAW/NV12** 原始格式，导致取流失败。
   - **尝试与结论**：尝试 FFmpeg 软编，因 CPU 占用率达 100% 且延迟过高（>5s）宣告失败。决定转向 **GStreamer + MediaMTX** 架构。

**#2026-02-25**
**终结 WebRTC 绿屏，实现“硬编 + 管道 + TCP”极限推流架构**
   - **遇到的问题**：使用 GStreamer + UDP 推流时，画面在大动作或录像时频繁出现**大面积绿屏**。
   - **根源分析**：
     1. **UDP 内部溢出**：大动作产生巨型 I 帧，瞬间流量击穿了 Linux 内核本地环回口的 UDP 接收缓冲区，导致数据包在板内丢失。
     2. **系统阉割**：板载 GStreamer 为精简版，缺失 `rtspclientsink` 等所有 TCP 相关插件，无法直接切换 TCP 协议。
   - **解决方案：缝合怪推流架构**
     - **流程**：`GStreamer (MPP硬编)` -> `Linux Pipe (标准输出)` -> `FFmpeg (网络搬运)`。
     - **原理**：利用 GStreamer 独占 RK3568 的 **MPP 硬件编码器**（0 CPU 占用）压制 H.264 流，通过管道（Pipe）实时喂给 FFmpeg。利用 FFmpeg 完善的协议栈，通过 `-rtsp_transport tcp` 强制进行 **TCP 传输**。
   - **最终成果**：成功在受限系统环境下实现了 **1080P/30FPS 的零丢包、零绿屏高清推流**。
